# Epic: Геймификация для резидентов

**Версия:** 2.0  
**Автор:** Надежда Балыдина  
**Дата:** 02.06.2026  
**Статус:** Draft for product / tech review  
**Источник:** устное описание идеи продукта после встречи

## 1. Краткое резюме

Нужно подготовить к разработке первую итерацию фичи геймификации: резиденты получают баллы за сдачу домашних заданий, видят баланс в личном кабинете, могут смотреть лидерборд, получают напоминание после недели неактивности, а администратор может вручную начислять баллы за особые активности.

Главный принцип MVP: сделать прозрачную и надёжную систему начисления баллов без расширения в уровни, бейджи, скидки, списания и сложные игровые механики.

## 2. Цель фичи

Повысить вовлечённость резидентов через понятную систему поощрения за активность на платформе.

### Метрики успеха после релиза

| Метрика | Что измеряем | Комментарий |
|---|---|---|
| Homework submission rate | Доля резидентов, сдающих ДЗ | Основная метрика вовлечённости |
| Weekly active residents | Количество активных резидентов за неделю | Проверяет влияние баллов и push |
| Return after inactivity push | Доля вернувшихся после уведомления | Проверяет US-4 |
| Leaderboard views | Количество просмотров лидерборда | Проверяет интерес к соревновательной механике |
| Manual accrual count | Количество ручных начислений | Проверяет, не заменяет ли ручной процесс автоматический |

## 3. Принятые допущения

Пока продукт не ответил на открытые вопросы, документ фиксирует рабочие допущения для оценки и планирования:

- Под «опозданием на день» понимается интервал от 1 секунды до 24 часов после дедлайна.
- Дедлайн сравнивается по серверному времени в UTC.
- При опоздании более 24 часов создаётся запись начисления с 0 баллов, чтобы сохранить прозрачный аудит.
- В лидерборде используется текущий баланс баллов.
- В MVP нет списания баллов, штрафов, уровней, бейджей и скидок.
- Push-уведомление в текущей итерации трактуется как web push или существующий канал уведомлений платформы; мобильный push требует отдельного подтверждения.
- Пользовательская история начислений не входит в MVP, но системный audit log обязателен для надёжности начислений.

## 4. Out of Scope

В текущую итерацию не входит:

- уровни, звания, бейджи и достижения;
- пользовательская история начислений в личном кабинете;
- анимации и визуальные эффекты при начислении баллов;
- списание, штрафные баллы и отрицательные баллы;
- интеграция баллов с оплатой, скидками или бонусами;
- A/B-тестирование механик геймификации;
- настройка псевдонима в профиле;
- мобильные push-уведомления, если у платформы ещё нет готового мобильного канала;
- антифрод-аналитика и модерация подозрительной активности;
- публичный профиль резидента.

## 5. MVP-состав и приоритеты

| ID | Задача | Приоритет | Почему |
|---|---|---|---|
| US-1 | Начисление баллов за сдачу ДЗ | High | Основа всей фичи |
| US-2 | Отображение баланса в ЛК | High | Без отображения пользователь не видит ценности |
| US-3 | Лидерборд | Medium | Усиливает мотивацию, но не блокирует базовую систему баллов |
| US-4 | Push при недельной неактивности | Medium | Влияет на retention, но технически независим от баллов |
| US-5 | Ручное начисление администратором | Medium | Нужно для особых случаев, но может идти после автоматического начисления |

## 6. Карта зависимостей

```text
[US-1] Начисление баллов
   ├── [US-2] Баланс в личном кабинете
   │       └── [US-3] Лидерборд: frontend-вход и UX-связь
   └── [US-5] Ручное начисление через единую точку записи

[US-4] Push при неактивности
   └── зависит от last_login_at и инфраструктуры уведомлений, но не от баллов
```

Важно: backend-часть лидерборда может проектироваться после US-1. Полная пользовательская поставка US-3 зависит от US-2, потому что пользователю сначала нужно видеть собственный баланс.

## 7. Definition of Ready

Задача готова к взятию в спринт, если:

- закрыты открытые вопросы, которые блокируют конкретную задачу;
- понятны роли пользователей и права доступа;
- есть критерии приёмки без двусмысленностей;
- определены зависимости от backend, frontend, notification service и admin panel;
- задача оценена командой;
- согласованы макеты или допустимо использовать текущие UI-компоненты платформы;
- для API-задач понятны входные и выходные данные;
- для задач с персональными данными согласованы ограничения приватности.

## 8. Definition of Done для всего Epic

Epic считается завершённым, если:

- все принятые user stories реализованы и прошли QA;
- все API задокументированы в OpenAPI / Swagger или внутреннем API-контракте;
- начисления баллов идемпотентны и не создают дублей;
- все изменения баллов пишутся в audit log;
- права доступа проверены для резидента, администратора и неавторизованного пользователя;
- основные сценарии покрыты unit / integration / e2e-тестами по решению команды;
- есть мониторинг ошибок начисления и отправки уведомлений;
- продукт принял тексты, отображение и поведение фичи;
- релиз сопровождается коротким release note для поддержки.

## 9. Non-Functional Requirements

| Требование | Ожидаемое поведение |
|---|---|
| Производительность баланса | Запрос текущего баланса — до 200 мс при штатной нагрузке |
| Производительность лидерборда | Страница лидерборда — до 500 мс при использовании кэша |
| Кэш лидерборда | Допустим кэш до 1 часа |
| Масштабируемость | Решение должно выдерживать 10 000 резидентов без смены архитектуры |
| Идемпотентность | Повторный запрос с тем же `submission_id` или `request_id` не создаёт дубль |
| Безопасность | Все пользовательские эндпоинты требуют авторизации |
| Admin security | Ручное начисление доступно только роли Administrator |
| Аудит | Все мутации баллов пишутся в append-only log |
| Наблюдаемость | Ошибки начисления, чтения баланса и отправки push логируются |
| Приватность | Баланс виден только владельцу и администраторам с нужной ролью |

## 10. Минимальная модель данных

Это не финальная техническая схема, а PM-уровень контракта, чтобы команда одинаково понимала сущности.

### `points_balance`

| Поле | Описание |
|---|---|
| `user_id` | ID резидента |
| `current_points` | Текущий баланс |
| `updated_at` | Дата последнего изменения |

### `points_ledger`

| Поле | Описание |
|---|---|
| `id` | ID записи |
| `user_id` | ID резидента |
| `points` | Количество начисленных баллов, включая 0 для прозрачного аудита |
| `source` | `homework_submission` / `manual_admin` |
| `reason` | Причина начисления |
| `submission_id` | ID сдачи ДЗ, если применимо |
| `request_id` | ID операции для идемпотентности |
| `created_by` | System или admin_id |
| `created_at` | Дата создания записи |

### `user_activity`

| Поле | Описание |
|---|---|
| `user_id` | ID резидента |
| `last_login_at` | Последний вход на платформу |
| `inactivity_notification_sent` | Было ли уже отправлено уведомление за текущую серию отсутствия |
| `notification_opt_out` | Отключил ли пользователь этот тип уведомлений |

## 11. User Stories

### US-1. Начисление баллов за сдачу домашнего задания

**Тип:** Backend  
**Приоритет:** High  
**Зависимости:** нет  
**Блокирует:** US-2, US-5, частично US-3

#### User Story

Как резидент, я хочу получать баллы за сдачу домашних заданий, чтобы видеть ощутимый результат своей активности на платформе.

#### Бизнес-правила

| Условие сдачи | Баллы |
|---|---:|
| До дедлайна включительно | 10 |
| После дедлайна, но не позднее 24 часов | 5 |
| Более 24 часов после дедлайна | 0 |

#### Acceptance Criteria

- При сдаче ДЗ до дедлайна включительно начисляется 10 баллов.
- При сдаче ДЗ после дедлайна, но не позднее 24 часов, начисляется 5 баллов.
- При сдаче ДЗ более чем через 24 часа после дедлайна создаётся запись с 0 баллов.
- Повторный вызов обработки одной и той же сдачи ДЗ не создаёт повторного начисления.
- Повторная отправка уже оценённого задания не пересчитывает ранее начисленные баллы без отдельного бизнес-решения.
- Если преподаватель изменил дедлайн после факта сдачи, уже начисленные баллы не пересчитываются автоматически.
- Каждая операция записывается в `points_ledger`.
- Баланс пользователя обновляется атомарно вместе с записью в ledger.
- При ошибке начисления система возвращает понятную ошибку и не создаёт частично записанное состояние.
- API или внутренний сервис начисления покрыт тестами на дедлайн, опоздание, повторный вызов и ошибочные данные.

#### Edge cases

- Сдача ровно в момент дедлайна считается своевременной.
- Сдача ровно через 24 часа после дедлайна даёт 5 баллов.
- Сдача через 24 часа и 1 секунду даёт 0 баллов.
- Если `deadline_at` отсутствует, начисление не выполняется, задача уходит в ошибку данных.

### US-2. Отображение баланса в личном кабинете

**Тип:** Frontend + Backend API  
**Приоритет:** High  
**Зависимости:** US-1  
**Блокирует:** полноценный UX для US-3

#### User Story

Как резидент, я хочу видеть текущий баланс баллов в личном кабинете, чтобы отслеживать прогресс без лишних действий.

#### Acceptance Criteria

- В личном кабинете отображается текущий баланс баллов резидента.
- При нулевом балансе отображается `0`, а не пустое поле.
- Баланс виден только владельцу аккаунта и администраторам с нужной ролью.
- Баланс обновляется не позднее чем через 30 секунд после начисления без перелогина.
- Если баланс не загрузился, пользователь видит нейтральное сообщение: «Не удалось загрузить баллы».
- Компонент корректно отображается на ширине экрана от 320px.
- Рядом с балансом может быть неактивная ссылка «История начислений» с меткой «Скоро», если продукт подтверждает такой UX.
- Endpoint баланса не отдаёт данные другого пользователя при подмене `user_id`.

#### Notes

Пользовательская история начислений не входит в MVP. Если ссылка «История начислений» вызывает лишние ожидания, лучше не показывать её в первой версии.

### US-3. Лидерборд резидентов

**Тип:** Frontend + Backend  
**Приоритет:** Medium  
**Зависимости:** US-1; UX-зависимость от US-2  
**Блокирует:** —

#### User Story

Как резидент с балансом больше 100 баллов, я хочу видеть своё место среди активных участников, чтобы поддерживать мотивацию.

Как резидент с балансом 100 баллов или меньше, я хочу видеть лидерборд и понимать, сколько баллов мне не хватает, чтобы попасть в рейтинг.

#### Правила участия

| Условие | Поведение |
|---|---|
| Баланс > 100 | Резидент отображается в лидерборде |
| Баланс ≤ 100 | Резидент видит лидерборд, но не отображается в нём |

#### Acceptance Criteria

- Лидерборд доступен только авторизованным пользователям.
- В лидерборде отображаются только резиденты с балансом больше 100 баллов.
- Участники сортируются по убыванию баллов.
- При равенстве баллов сортировка выполняется по имени или фамилии по алфавиту.
- Для каждого участника отображается место, имя / псевдоним и количество баллов.
- Резидент с балансом 100 или меньше видит сообщение: «До лидерборда осталось N баллов».
- Если участников нет, отображается заглушка: «Пока здесь пусто — стань первым!».
- Лидерборд обновляется не реже одного раза в час.
- При большом количестве участников используется пагинация или другой согласованный способ ограничения выдачи.
- Неавторизованный пользователь перенаправляется на страницу входа.

#### Privacy note

До старта US-3 продукт должен подтвердить, что показываем: полное имя, имя + первая буква фамилии или псевдоним.

### US-4. Push-уведомление при недельном отсутствии

**Тип:** Backend + Notifications  
**Приоритет:** Medium  
**Зависимости:** `last_login_at`, notification service  
**Блокирует:** —

#### User Story

Как платформа, я хочу напоминать резиденту после 7 дней отсутствия, чтобы снизить риск оттока и вернуть его к обучению.

#### Текст уведомления

Черновик: «Эй, ты теряешь темп! Зайди на платформу и продолжи обучение.»

Финальный текст должен быть согласован с продуктом или копирайтером до релиза.

#### Acceptance Criteria

- Система обновляет `last_login_at` при каждом входе резидента.
- Ежедневная задача выбирает резидентов, которые не заходили 7 или более дней.
- Уведомление отправляется не более одного раза за одну непрерывную серию отсутствия.
- После входа резидента флаг отправки сбрасывается.
- Баллы не сгорают и не изменяются из-за отсутствия.
- Если пользователь отключил этот тип уведомлений, push не отправляется.
- Каждая попытка отправки логируется с `user_id`, `sent_at`, `channel`, `status`.
- Повторный запуск cron-job не создаёт дубль уведомления одному и тому же пользователю.
- При недоступности notification service ошибка логируется, но не ломает другие процессы платформы.

### US-5. Ручное начисление баллов администратором

**Тип:** Admin Panel + Backend  
**Приоритет:** Medium  
**Зависимости:** US-1 / единый сервис записи баллов  
**Блокирует:** —

#### User Story

Как администратор, я хочу вручную начислять баллы резиденту, чтобы поощрять активность, которую система не отслеживает автоматически.

#### Acceptance Criteria

- В admin panel доступна функция «Начислить баллы вручную».
- Функция доступна только пользователям с ролью Administrator.
- Администратор может найти резидента по имени или email.
- Поиск запускается минимум после 3 введённых символов.
- Поле «Количество баллов» принимает целое число от 1 до 500.
- Поле «Причина» обязательно и принимает 5–200 символов.
- Перед сохранением отображается confirmation modal с пользователем, количеством баллов и причиной.
- После подтверждения баллы начисляются через тот же backend-сервис, что и автоматическое начисление.
- Операция записывается в `points_ledger` с `source = manual_admin` и `created_by = admin_id`.
- Администратор видит последние 10 ручных начислений, чтобы снизить риск случайного дубля.
- Пользователь видит обновлённый баланс согласно US-2.
- Пользователь без роли Administrator получает `403 Forbidden`.

#### Important constraint

US-5 в MVP — это только положительное ручное начисление. Отрицательные корректировки и отмена ошибочных операций требуют отдельного решения продукта и Tech Lead.

## 13. Риски и способы снижения

| Риск | Последствие | Как снижаем |
|---|---|---|
| Дубли начислений | Недоверие к системе баллов | Идемпотентность, ledger, тесты на повторный вызов |
| Неясные дедлайны и timezone | Споры с пользователями | Фиксируем правило времени до разработки |
| Лидерборд демотивирует новичков | Снижение вовлечённости | Порог >100 и сообщение «до лидерборда осталось N» |
| Нарушение приватности в лидерборде | Жалобы пользователей | Согласовать формат имени / псевдонима |
| Push воспринимается агрессивно | Негатив к платформе | Согласовать текст и дать возможность отключить уведомления |
| Ручное начисление используют вместо системного | Рост ручной операционки | Логирование и мониторинг количества manual accruals |
| Нет процедуры исправления ошибок | Сложность поддержки | Отдельное решение по correction flow |

## 14. Рекомендуемый порядок разработки

| Волна | Задачи | Условие старта | Результат |
|---|---|---|---|
| 1 | US-1 | Закрыты вопросы 1–2 | Надёжное начисление и ledger |
| 1 параллельно | US-4 | Есть `last_login_at` и notification channel | Retention-механика без зависимости от баллов |
| 2 | US-2 | Принята backend-логика US-1 | Пользователь видит баланс |
| 2 | US-5 | Есть единый сервис записи баллов | Админ может начислять баллы вручную |

## 16. Процесс выполнения задания и использование AI

**Вариант:** использован AI / LLM как ассистент для структурирования, аудита и самопроверки.

**Модель:** ChatGPT GPT-5.5 Thinking.  
**Почему выбрана:** модель хорошо подходит для анализа неструктурированного текста, поиска противоречий, формализации user stories, критериев приёмки, рисков и зависимостей.

### Промпты, которые использовались

#### Prompt 1 — первичная структуризация

```text
Ты опытный Project Manager в продуктовой IT-команде.
Ниже есть устное описание фичи от продакта.
Нужно превратить его в готовые задачи для разработки:
- выделить epic и user stories;
- прописать acceptance criteria;
- указать зависимости;
- найти противоречия и открытые вопросы;
- отделить MVP от out of scope;
- сделать документ понятным для разработки, QA и продукта.
```

#### Prompt 2 — экспертный аудит

```text
Посмотри на документ как эксперт PM.
Проверь, чего не хватает для передачи задач в спринт.
Найди противоречия, слабые критерии приёмки, неверные зависимости,
скрытые технические риски и места, где разработка может трактовать задачу по-разному.
Предложи исправления.
```

#### Prompt 3 — финальная самопроверка

```text
Перепроверь документ ещё раз как строгий reviewer.
Ищи несоответствия между scope, acceptance criteria, зависимостями,
open questions, DoR, DoD и порядком разработки.
Особенно проверь, нет ли противоречий между ручным начислением,
корректировками, историей начислений, audit log и out of scope.
```

**Чистое время выполнения:** около 1,5–2 часов: чтение задания, первичная структура, экспертная доработка, самопроверка и финальная сборка документа.
